التقرير الشامل لمشروع طلب وتوصيل أسطوانات الغاز

هذا التقرير يلخص حالة المشروع الحالية، ما تم إنجازه فعليًا، ما يحتاج تعديلًا، النواقص الموجودة، ووضع خطة عمل عملية مقسمة على كل قسم من أقسام النظام: تطبيق العميل، تطبيق السائق، الـBackend ولوحة التحكم، قاعدة البيانات، وطبقة التشغيل عبر Docker.

جاهز للتجربة المحلية Flutter Node.js + Express Next.js Dashboard PostgreSQL Socket.IO

تقسيم النظام

أقسام المشروع وحالة كل قسم

1. تطبيق العميل

مكتمل جزئيًا

مبني بـ Flutter ويغطي المصادقة، عرض المنتجات، إنشاء الطلب، الطلبات السابقة، وخريطة التتبع.

  • إنشاء الطلب وربطه مع API فعليًا تم.
  • قراءة موقع الجهاز الفعلي مع fallback للمحاكي تم.
  • تحديثات الطلب الحية موجودة في الـstate.
  • شاشة تتبع الطلب ما زالت تعرض موقع الجهاز بدل تتبع الطلب الحقيقي.

2. تطبيق السائق

مكتمل جزئيًا

مبني بـ Flutter كذلك، ويغطي الطلبات المخصصة للسائق، مشاركة الموقع، وخريطة التسليم النشطة.

  • إرسال موقع السائق إلى الخادم تم.
  • إظهار موقع العميل والسائق على الخريطة تم.
  • تهيئة Google Maps على Android تم تحسينها.
  • المسار الحالي خط مستقيم وليس مسار طرق حقيقي.

3. الـBackend ولوحة الإدارة

الأساس موجود

الخادم يشغل Express API و Next.js Dashboard و Socket.IO على نفس الـserver.

  • الطلبات والمصادقة والمنتجات والمناطق والإعدادات موجودة.
  • لوحة الإدارة تحتوي صفحات dashboard و orders و drivers و zones.
  • البث اللحظي للأحداث موجود عبر Socket.IO.
  • هناك حاجة لتشديد العقد والـvalidation والمراقبة التشغيلية.

4. قاعدة البيانات والتشغيل

تعمل محليًا

PostgreSQL تعمل عبر Docker مع ملف تهيئة أولي، وpgAdmin متاح للإدارة البصرية.

  • الجداول الأساسية موجودة مع seed data أولي.
  • فهارس أساسية موجودة للطلبات والعملاء والسائقين.
  • هناك ازدواج في حقول الإحداثيات داخل orders.
  • لا توجد بعد طبقة migrations وإدارة نسخ احتياطي احترافية.

ما تم إنجازه

الأعمال المنفذة حتى الآن

البنية الأساسية والتشغيل

  • إعداد Docker Compose لتشغيل backend و postgres و pgAdmin.
  • تجهيز Dockerfile لبناء وتشغيل Next.js و Node.js.
  • ربط متغيرات البيئة الأساسية للمنافذ وبيانات PostgreSQL.

الخادم والـAPI

  • بناء Express app وربطه مع Next handler داخل نفس الخادم.
  • تجهيز مسارات API: auth و driver-auth و driver و orders و drivers و customers و settings و products و zones.
  • إعداد بث حي للأحداث المتعلقة بالطلبات والسائقين.

قاعدة البيانات

  • إنشاء جداول العملاء والسائقين والطلبات والإعدادات والمنتجات والمناطق.
  • إضافة بيانات أولية للنظام والمنتجات ومناطق التوصيل.
  • إضافة فهارس لتحسين الاستعلامات الأساسية.

تطبيق العميل

  • ربط إنشاء الطلب الحقيقي مع `/api/orders`.
  • جلب الموقع الدقيق عبر geolocator و geocoding.
  • إضافة fallback خاص بالمحاكي داخل مسقط للاختبار.
  • تهيئة Google Maps قبل تشغيل التطبيق.
  • إصلاح مشكلة layout في شريط فلاتر الطلبات داخل شاشة طلباتي.

تطبيق السائق

  • مشاركة موقع السائق الحي أثناء التوصيل.
  • عرض موقع العميل وموقع السائق في شاشة التسليم النشط.
  • تهيئة Android renderer للخرائط مع warmup.
  • دعم قراءة الإحداثيات من أكثر من شكل payload في نموذج الطلب.

خدمة الموقع والخرائط

تقييم وضع خدمة الـ Map حاليًا

خدمة الموقع موجودة ومفيدة بالفعل، لكنّها ليست مكتملة بعد كتجربة تتبع تشغيلية كاملة من العميل إلى السائق إلى لوحة الإدارة.

ما الذي يعمل الآن

  • العميل يلتقط موقعه ويرسل الإحداثيات عند إنشاء الطلب.
  • الخادم يستقبل الإحداثيات ويحفظها داخل الطلب.
  • السائق يرى موقع العميل على الخريطة.
  • السائق يرسل موقعه الحالي إلى الخادم أثناء التسليم.
  • تطبيق العميل يستطيع استقبال تحديثات موقع السائق في الـstate.

أكبر نقص حالي

  • شاشة تتبع الطلب عند العميل لا تعرض الطلب الحقيقي بعد.
  • المسار في تطبيق السائق ليس route فعلي من الطرق.
  • حقول الإحداثيات في قاعدة البيانات مزدوجة وتحتاج توحيدًا.
  • مفاتيح Google Maps ما زالت تحتاج إنهاء صارم للإنتاج.

الاقتراح الفني

  • الاستمرار مع Google Maps Platform كخيار أساسي.
  • إضافة Directions أو Routes API عند الانتقال لمسارات حقيقية.
  • توحيد عقد الإحداثيات على صيغة واحدة في الـAPI والـDB.
  • تحويل شاشة تتبع العميل لاستخدام orderById بدل موقع الجهاز.
  • إضافة مناطق توصيل دقيقة لاحقًا بدل فحص حدود عمان العامة.

النواقص الحالية

ما الذي يحتاج تعديلًا أو إنهاءً

تطبيق العميل

  • ربط شاشة تتبع الطلب بالبيانات الحقيقية بدل موقع الجهاز الحالي.
  • تقوية تجربة إدارة العناوين والاعتماد على بيانات المناطق.
  • مراجعة fallback الخاص بالمحاكي حتى لا يختلط مع السلوك الإنتاجي.

تطبيق السائق

  • استبدال الـpolyline المستقيم بمسار حقيقي.
  • تحسين إدارة مشاركة الموقع عند تغير lifecycle أو انقطاع الشبكة.
  • تجهيز مفاتيح الخرائط للإنتاج بشكل آمن.

الـBackend

  • توحيد contract الإحداثيات والحقول الجغرافية.
  • تشديد التحقق من المدخلات والـauthorization في جميع المسارات.
  • إضافة اختبارات API ووثائق أوضح للعقود بين التطبيقات.

لوحة الإدارة

  • تعميق الربط التشغيلي مع حالات الطلبات والسائقين لحظيًا.
  • إضافة مؤشرات تشغيل حقيقية وتقارير يومية واضحة.
  • إظهار خرائط أو مواقع الطلبات داخل الواجهة الإدارية عند الحاجة.

4. التقارير والرقابة

  • تقارير يومية وأسبوعية عن الطلبات والإيرادات وأداء السائقين.
  • سجل عمليات audit log يوضح من غيّر ماذا ومتى.
  • لوحة KPI أوضح للإدارة العليا.
  • تصدير CSV أو Excel للطلبات والعملاء والسائقين.

قاعدة البيانات

  • إزالة الازدواج بين latitude/longitude و customer_latitude/customer_longitude.
  • اعتماد migrations منظمة بدل الاكتفاء بملف init واحد.
  • وضع سياسة backup و restore واختبارات بيانات.

DevOps والجودة

  • لا توجد بعد pipeline CI/CD كاملة.
  • المراقبة والـlogging والتنبيهات ليست مكتملة.
  • إعدادات الإنتاج وHTTPS وإدارة الأسرار تحتاج إنهاء.

خطة العمل المقترحة

خارطة تنفيذ عملية مقسمة حسب كل قسم

قسم تطبيق العميل

  1. ربط شاشة تتبع الطلب ببيانات الطلب الفعلية وموقع السائق الحي.
  2. تحسين شاشة العنوان الحالي وتجربة اختيار العنوان.
  3. مراجعة حالات الفشل في الشبكة والموقع والخرائط.
  4. إضافة اختبارات widget وintegration للمسارات الحرجة.

قسم تطبيق السائق

  1. إضافة route حقيقي و ETA أدق داخل شاشة التسليم.
  2. تحسين استقرار مشاركة الموقع وإعادة الربط عند الانقطاع.
  3. إضافة تجربة أوضح لحالات رفض الإذن أو توقف GPS.
  4. إقفال ملف مفاتيح الخرائط وتجهيز build للإنتاج.

قسم الـBackend والـAPI

  1. توحيد payload الإحداثيات بشكل رسمي داخل order contract.
  2. إضافة schema validation واضحة لكل endpoint مهم.
  3. تحسين توثيق الأحداث اللحظية وقواعد تحديث حالة الطلب.
  4. إضافة اختبارات وحدات واختبارات API آلية.

قسم لوحة الإدارة

  1. استكمال صفحات الإدارة التشغيلية وربطها ببيانات حية.
  2. إضافة أدوات متابعة للطلبات والسائقين والمناطق.
  3. إظهار تقارير تشغيلية ورسوم بيانية ومؤشرات أداء.
  4. إضافة صلاحيات أدوار إدارية إن كان النظام سيكبر لاحقًا.

قسم قاعدة البيانات

  1. تنظيف schema الطلبات وتوحيد الحقول الجغرافية.
  2. إدخال migrations رسمية وفهارس إضافية حسب الاستعلامات الفعلية.
  3. فصل بيانات الإنتاج عن بيانات seed والتجارب.
  4. إعداد سياسة backup دورية واستعادة.

قسم التشغيل والجودة

  1. إعداد بيئات development و staging و production بوضوح.
  2. إضافة CI للبناء والتحليل والاختبارات.
  3. إضافة logging و monitoring و alerts.
  4. إغلاق الأسرار والمفاتيح وضبط HTTPS والنشر.

الخلاصة

التقييم النهائي

المشروع الآن يملك أساسًا قويًا: تطبيق عميل يرسل الطلبات، تطبيق سائق يتابع التسليم، Backend موحد مع لوحة إدارة، وقاعدة بيانات جاهزة محليًا. أكبر أولوية عملية تالية هي إكمال تجربة التتبع عند العميل وتوحيد عقد الإحداثيات، ثم الانتقال إلى تحسين الجودة التشغيلية والاختبارات وبيئة الإنتاج.

باختصار: ما تم إنجازه جيد وقابل للبناء عليه، لكن قبل الإطلاق الرسمي ما زال هناك عمل واضح ومحدد، والخطة الموجودة في هذا التقرير تصلح كخارطة تنفيذ مباشرة للمرحلة القادمة.
